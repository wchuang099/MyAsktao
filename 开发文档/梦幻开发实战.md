## 1. 核心架构

### 1.2 主控与被控的分离设计

在高级脚本开发中，**“主控机（脚本端）+ 被控机（游戏端/虚拟机）”**的架构是实现高隐蔽性与跨环境操作的核心。

* **物理隔离**：通过虚拟机（VM）承载游戏，主控机承载脚本。这种环境隔离可有效规避游戏客户端对脚本进程的直接检测。
* **交互逻辑**：
  * **画面捕获**：利用**VNC**协议，将虚拟机的实时画面流式传输至主控机的 UI 窗口。
  * **指令反馈**：脚本通过大漠插件（或其他图色工具）绑定 VNC客户端窗口进行图像识别，并将键鼠信号通过内网协议回传至虚拟机执行。

> **架构优势：为何不直接绑定虚拟机？**
>
> - 虚拟机属于独立系统，直接后台绑定往往无法正常发送键鼠指令（除非通过双头硬件或自写通信端）。
> - VNC客户端是一个标准化的句柄窗口，方便大漠插件进行后台绑定与操作

### 1.1 常见技术方案对比

根据成本、隐蔽性及实现难度，可分为以下三种主流方案：
| 方案名称                    | 技术实现路径                                                 | 核心优势                                                     | 潜在挑战                                                     |
| --------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 方案 1：VNC 标准协议流      | **1-1  VNC**：开启VMware 自带 VNC 服务端，宿主机通过 VNC 客户端连接。<br>**1-2 自研 VNC 插件**（类似unify）：不通过vnc客户端，插件直接连接vnc服务端，图色获取，指令发送 | 低成本、易落地。技术成熟稳定，适合快速部署和规模化使用，是目前最通用方案。 | 对网络带宽有一定要求，在极端情况下画面传输可能存在毫秒级延迟。 |
| 方案 2：虚拟机 + 物理硬件   | 大漠插件负责 VM 窗口画面识别，操作指令通过 **双头盒子（USB HID）硬件设备**发送到虚拟机。 | 极致安全性。模拟真实物理键盘鼠标信号，几乎等同真人操作，理论上 **零封号风险**。 | 硬件成本高，每台虚拟机需配备独立硬件设备                     |
| 方案 3：图色识别 + 自研协议 | **宿主机**：大漠插件进行图像识别。<br>**虚拟机**：部署自研 Agent 通信程序。<br>**逻辑**：宿主机通过自定义协议发送指令，Agent 调用 `SendInput` 执行操作。 | 高扩展性、零硬件成本。逻辑控制灵活，支持复杂自动化与自定义指令。 | 隐蔽性存在风险，需要在虚拟机内运行 Agent 进程，可能被安全检测扫描。 |

### 1.3 虚拟机环境设置

1. **版本选择**

* **推荐版本**：VMware Workstation **15.x 及以下版本**。
* **规避风险**：16.x 及以上版本在进行大漠插件后台绑定时，极易出现**画面黑屏**的情况。
* **检查方法**：`帮助` -> `关于 VMware Workstation` 查看具体版本号。

2. **VNC 被控端配置**

虚拟机自带 VNC 功能（注意：部分魔改精简版可能阉割了此功能，需自行安装vnc）。
* **设置路径**：右键虚拟机 -> `设置` -> `选项` -> `VNC 连接`。
* **关键参数**：
  * **启用 VNC 连接**：勾选。
  * **端口**：建议自定义（如 `5901`），避免多开冲突。
  * **密码**：必须设置（如 `123`），用于远程连接鉴权。


### 1.4 VNC 远程连接与连接异常处理

1. 连接格式

在主控机使用 VNC Viewer 访问，格式为：`IP:端口`。

* **本机访问**：使用回环地址 `127.0.0.1:5901`。
* **注意**：冒号必须使用**英文输入法**下的半角冒号 `:`

2. 初次连接蓝屏/不刷新处理

若连接后画面不显示或操作无响应（预览模式故障）：

1. 调整 VNC Viewer 的**画质设置**（Quality）。

2. 关闭连接窗口并重新启动。


### 1.5 核心知识点：画质统一与系统设定

> **注意**：很多开发者遇到“本地运行正常，客户机器跑不起来”的问题，90% 都是因为**色差**和**系统主题**不统一。

1. VNC 画质设定（至关重要）

* **规则**：**严禁使用“自动（Automatic）”画质**。
* **原因**：自动模式会根据内网带宽波动自动调节。画质改变会导致画面产生位深度压缩和色差。
* **后果**：原本截好的图片（图库）或定义的颜色值，在画质改变后将无法识别（大漠找图找色失败）。
* **要求**：开发与用户端必须统一设定为 **High Quality（高画质）**。

2. 被控端系统主题统一

- **原理**：Windows 主题（如 Aero 特效、经典主题）会改变窗口标题栏、按钮、文字的颜色和渲染方式。
- **风险**：
  - **五色字**：在非经典主题下，原本单色的文字可能变成带有渐变、阴影的“五色字”，极大增加字库制作难度，甚至无法识别。
  - **窗口元素变化**：标题栏高度、边框样式等发生变化，影响基于坐标的点击操作。
- **操作**：
  1. **强制统一**：要求所有被控端（虚拟机或物理机）将系统主题设置为 **“Windows 经典”** 模式。
  2. **备用方案**：如果无法更改主题，则要求用户将系统性能选项调整为 **“调整为最佳性能”**。此操作会关闭大部分视觉特效，使界面回归类似经典主题的简洁样式。
     - **路径**：计算机（右键）-> 属性 -> 高级系统设置 -> 高级（选项卡）-> 性能（设置）-> 视觉效果 -> 调整为最佳性能。

3. 分辨率与窗口坐标

* **坐标参考**：建议开发时将游戏窗口固定在屏幕的 `(0, 0)` 位置。
* **兼容性说明**：只要坐标系和窗口相对位置固定，分辨率的轻微改变通常不会影响基于偏移量的识别逻辑。

### 1.6  物理机作为被控端的配置

若被控机是真实物理电脑，需手动安装 VNC Server：

1. **安装选项**：选择注册为系统服务（Service模式）。
2. **必做设定**：
* **设置密码**：在 `Security` 选项中设置 VNC 密码。
  
* **关闭闲时断开**：将 `IdleTimeout`（闲时超时）设置为 `0`（代表禁用），防止因长时间不操作导致连接中断。
  
* **固定 IP**：在路由器或网卡设置中手动固定内网 IP 地址，防止 IP 漂移导致连接失效。





## 2. 易语言UI 构建：中控台设计

脚本需提供良好的交互界面，核心控件为 **“超级列表框”**。

### 2.1 控件初始化

- **加载扩展组件**：若组件栏无超级列表框，需通过 `工具 -> 知识库 -> 全选` 加载扩展组件。
- **报表模式设定**：
  - 将超级列表框的 `类型` 属性修改为 **“报表列表框”**。
  - 勾选属性：**“整行选择”** 与 **“显示表格线”**。
- **视觉优化**：通过调整 `字体`（如微软雅黑，10-12号）来自动撑开行高，提升易用性。

### 2.2 表头（报表列）定义

常用字段包括：`索引`、`vnc窗口名称`、`账号`、`等级`、`当前任务`、`运行状态`。

### 2.3 右键菜单与交互逻辑

**1.弹出菜单实现**

- 菜单编辑器：创建主菜单（设为不可见）及其子菜单（如：功能设置）。

- 事件触发：

  - 监听超级列表框的 **“右键单击表项”** 事件。

  - 避坑指南：必须判断 `现行选中项 != -1`，防止用户点击表头或空白处时弹出无效菜单。

  - 使用 `弹出菜单()` 指令呼出菜单。

**2.窗口载入模式**

- 关键参数：载入功能设置窗口时，第四个参数（类型）务必设为 **`真` (Modal 模式)**。
- 原因：强制用户必须处理完当前设置窗口后才能操作主界面，防止逻辑混乱或多窗口覆盖导致的参数配置错误。

### 2.4  单窗口配置隔离（多线程基础）

当脚本支持多开时，必须解决“全局配置冲突”问题。如果所有窗口都读写同一个 INI 节点，会导致参数覆盖。

**1. 索引映射机制**

引入一个全局变量 `现行选中行_索引` (整数型)。

- 当用户在超级列表框右键点击时，获取当前行索引：`索引 = 超级列表框.现行选中项`。
- 为了符合用户直觉，通常将内部索引（从0开始）转换为显示索引（从1开始）：`全局变量 = 索引 + 1`。

**2. 动态节点配置读写**

利用 INI 文件的 **“节点 (Section)”** 机制实现数据隔离。

- **写入配置**：

  代码段

  ```
  写配置项 (运行目录 + "config.ini", "窗口_" + 到文本 (全局变量), "VNC名称", 编辑框_VNC.内容)
  ```

- **读取配置**：

  代码段

  ```
  编辑框_VNC.内容 = 读配置项 (运行目录 + "config.ini", "窗口_" + 到文本 (全局变量), "VNC名称", "默认窗口")
  ```

**3. 实现效果**

- **窗口1** 的设置保存在 `[窗口_1]` 节点下。
- **窗口2** 的设置保存在 `[窗口_2]` 节点下。
- **结果**：点击不同行打开设置界面时，程序根据 `全局变量` 动态定位节点，确保存储与读取的内容完全独立，互不干扰。

**4. 避坑小贴士**

1. **路径处理**：调用 `取运行目录()` 后，务必手动加上路径分隔符 `\`，否则会导致文件路径拼接错误，无法生成配置文件。
2. **默认值设定**：`读配置项` 的第四个参数应填入合理的默认值，防止用户初次打开脚本时参数为空导致程序崩溃。
3. **命名规范**：控件命名建议加前缀（如 `编辑框_VNC名称`），避免工程量变大后逻辑混淆。

## 3.大漠插件基础

### 3.1 大漠插件初始化流程

**1.系统注册控件**

大漠插件的调用必须严格遵循以下四步：
第一步：注册 COM 组件到系统

- 权限注意：如果注册返回 `假`（失败），通常是因为权限不足。必须以管理员身份运行易语言。
- 代码实现：调用 `注册组件 (运行目录 + "\资源\dm.dll")`。

第二步：创建大漠对象
- 全局变量：为了方便在不同程序集调用，建议将大漠对象声明为**全局变量**（类型：`大漠中文版` 或 `dm_soft`）。
- 代码：`大漠.创建对象 ()`。

第三步：获取版本号

- 验证对象是否创建成功。若返回为空文本，则插件初始化失败。

第四步：注册收费 VIP 插件（防止闪退）

- 重要性：大部分核心功能（后台绑定等）属于收费功能，不注册会导致脚本**运行中途闪退**。
- 命令：`大漠.Reg (注册码, 附加码)`。
  - `返回值 = 1`：注册成功。
  - `返回值 = -2`：权限问题，需管理员运行。

**2.大漠也支持免注册调用**
免注册调用的好处是**不向系统注册控件**，具有一定的防检测效果（隔离性好）。

- **准备工作**：从大漠后台下载免注册专用的 `dm.dll` 和 `DmReg.dll`，并放入脚本的资源目录。
- **调用流程**：
  1. 调用天野模块的“免注册大漠”命令。
  2. **参数1**：`DmReg.dll` 的路径。
  3. **参数2**：`dm.dll` 的路径。
  4. **返回值**：逻辑型（真为成功）。
- **注意**：免注册成功后，后续仍需执行“创建对象”和“注册VIP”的操作。

### 3.2  找图命令详解（FindPic）

1. 基础找图 `FindPic`

- **坐标系**：输入查找区域的左上角 (x1, y1) 和右下角 (x2, y2)。
- **图片路径**：如果在初始化时设置了 `大漠.SetPath (资源路径)`，则图片名无需再带长路径。
- **返回值机制**：
  - `-1`：未找到图片。
  - `0, 1, 2...`：代表找到了第几张图（对应输入参数中的图片索引）。
- **坐标返回**：通过“参考”参数（指针）返回找到图片的**左上角坐标**，而非中心点。

2. 多图查找 `FindPicEx`

- **适用场景**：屏幕上有多个相同目标（如多个文件夹图标）时，需要一次性获取所有位置。
- **返回格式**：返回一个长字符串，格式为 `图片索引,x,y|图片索引,x,y|...`。
- **解析方式**：需使用 `分割文本` 命令，先按 `|` 分割组，再按 `,` 分割具体坐标。



### 3.3 常见问题与技巧

1. **管理员权限**：这是 90% 注册失败的原因。右击易语言程序 -> 属性 -> 兼容性 -> 勾选以管理员身份运行。

2. **相对路径与绝对路径**：
   - `SetPath` 设置全局路径后，所有的 `找图`、`找字`、`找色` 均只需填写文件名（如 `1.bmp`）。
   - **避坑**：但在 `注册组件` 时，必须使用**绝对路径**（取运行目录 + 文件名）。
   
3. **找图偏差**：大漠找图默认返回的是图片的左上角。如果需要点击图片中心，需根据图片宽高自行进行坐标偏移（X + 宽/2, Y + 高/2）。

4. **调试输出**：在开发初期，务必在每一步初始化（注册系统、创建对象、注册VIP）后增加 `调试输出`，以便快速定位错误码。

   

### 3.4 坐标系深度解析

理解坐标系是写脚本不“翻车”的基础。

- **屏幕坐标（未绑定）**：
  - **起点 (0,0)**：位于屏幕的最左上角。
  - **方向**：X轴向右变大，Y轴向下变大。
- **窗口坐标（绑定后）**：
  - **起点 (0,0)**：变为**绑定窗口/客户区**的左上角。
  - **易错点**：很多新手在未绑定工具的情况下取坐标，结果绑定后坐标偏移，导致点击失败或“越标”报错。
- **越标报错**：如果点击或截图的坐标超出了窗口范围（例如窗口只有800x600，你却点1000x1000），大漠会报错。这通常是因为坐标计算错误或未区分前后台坐标导致的。

### 3.5 后台绑定

绑定是实现“后台挂机”的关键。

- **核心命令**：`dm.BindWindowEx(hwnd, display, mouse, keypad, public, mode)`
- **关键参数测试（使用综合工具）**：
  1. **句柄 (hwnd)**：窗口的身份证号，每次启动都会变。
  2. **画面 (display)**：
     - `Normal`：前台，不能遮挡，遮挡或移出屏幕会黑屏。
     - `GDI / DX`：后台模式，支持遮挡和移出屏幕（部分支持最小化）。
  3. **鼠标 (mouse) / 键盘 (keypad)**：
     - `Normal`：会抢占你真实的鼠标。
     - `Windows / DX`：后台模拟，鼠标可以放在窗口后面操作，不影响你用电脑。
  4. **公共属性 (public)**：
     - 常用：`dx.public.active.message`（后台激活）、`dx.public.disable.size`（禁止用户改变窗口大小）。

绑定技巧

- **绑定测试流程**：先用综合工具手动测试哪个模式能获取画面、能点击，测试通过后再写代码。
- **代码健壮性**：
  - 绑定前先判断窗口是否最小化，若是则先置顶并恢复大小。
  - 绑定后立即检查返回值，若不等于1则停止脚本并提示失败。
- **取点规范**：**一定要在工具绑定窗口的状态下取坐标！** 否则获取的是屏幕坐标，后台模式下无法直接使用。
- **窗口检测**：在绑定前检查目标窗口是否最小化（`窗口_是否最小化`），若最小化则强制恢复（`窗口_置状态 (12)`），否则会导致画面不刷新或绑定失败。
- **强制尺寸自适应**：无论用户如何拖拽，代码统一强制设置窗口客户区大小（如 `1040 * 807`），确保坐标计算的一致性。

## 4. 易语言多线程控制

### 1.1 自定义数据类型 (Structure/Struct)

在多线程开发中，自定义数据类型用于封装每个线程的独立状态，实现逻辑解耦。

- **声明方式**：`查看` -> `自定义数据类型表` -> `新增`。
- **结构示例**：
  - **类型名称**：`线程信息`
  - **成员**：
    - `线程句柄` (整数型)
    - `线程状态` (文本型：启动/停止/暂停)
    - `窗口句柄` (整数型)
    - `大漠对象` (对象/dm_soft)
- **引用方式**：声明全局数组变量，通过 `重定义数组` 动态匹配超级列表框的表项数，防止数组下标越界。

### 1.2 多线程任务管理架构

构建一套稳健的线程控制逻辑（启动、停止、暂停、恢复）。

- **初始化核心**：
  1. **动态数组初始化**：使用 `重定义数组 (信息数组, 假, 超级列表框.取表项数())` 实现线程信息与 UI 绑定。
  2. **COM 库初始化**：线程起始处必须调用 `CoInitialize (0)` 或大漠提供的 `线程_初始化COM库()`，确保对象实例化成功。
- **线程安全操作**：
  - 通过 `线程_启动 (子程序地址, 线程索引)` 传递索引值，让子程序内部通过索引访问对应的自定义数据成员。
  - **暂停/恢复**：利用 `线程_挂起()` 和 `线程_恢复()` 维护线程生命周期。





## 5. 鼠标漂移解决方案 (重点)

### 5.1 漂移成因分析

在某些 3D 或特殊引擎游戏（如梦幻西游）中，传统的后台绝对坐标移动（`MoveTo`）会因为客户端内部校验或模拟环境（VNC/虚拟机）的坐标转换差异，导致实际落点产生偏差。

### 5.2 方案一：基于字库查找的“相对位移校准法”

**实现逻辑步奏：**

1. **建立鼠标特征字库**：利用大漠综合工具，截取鼠标指针在不同背景下的二值化特征，存入专用字库。

2. **执行初始移动**：先调用 `MoveTo` 移动到目标坐标。

3. **实时位置检索**：调用 `FindString` 或 `FindPic` 在目标点附近寻找鼠标实际出现的坐标。

4. **计算误差值（Delta）**：

   - `误差X = 目标坐标X - 实际坐标X`
   - `误差Y = 目标坐标Y - 实际坐标Y`

5. **循环相对修正**：

   - 使用 `MoveR (误差X, 误差Y)` 执行相对移动。

   - 通过循环不断微调（Loop Correction），直至误差落入设定的精度范围（如 1~3 像素内）。

     

### 5.3 方案二：基于YOLO查找的“相对位移校准法”

**实现逻辑步奏：**

1. **训练鼠标游戏特征图片**

2. **执行初始移动**：先调用 `MoveTo` 移动到目标坐标。

3. **实时位置检索**： 在目标点附近寻找鼠标实际出现的坐标。

4. **计算误差值（Delta）**：
   - `误差X = 目标坐标X - 实际坐标X`
   - `误差Y = 目标坐标Y - 实际坐标Y`

5. **循环相对修正**：

   - 使用 `MoveR (误差X, 误差Y)` 执行相对移动。

   - 通过循环不断微调（Loop Correction），直至误差落入设定的精度范围（如 1~3 像素内）。





### 5.4注意事项

1. **字库精细度**：鼠标指针在闪烁或经过 NPC 时颜色会变，建议在不同场景下多取几组特征码，提高 `FindString` 的识别率。
2. **性能平衡**：`鼠标相对移动` 的步长不宜设置过大，否则会产生视觉上的抖动；过小则校准速度慢，建议根据机器性能动态调整 `延时` 参数。

3. **OpenCV 找图集成（备选方案）**:由于大漠字库在面对某些动态干扰或背景复杂的图片（如移动的鼠标针尖、带光效的NPC）时易失效，我们引入 OpenCV 模块作为补充。
- 核心逻辑：
  1. **先找字库**：利用大漠快速识别（性能损耗低）。
  2. **后补 OpenCV**：如果字库没找到，再截图并调用 OpenCV 找图（识别率极高，支持相似度/信度判定）。
- 注意事项：
  - **可信度（信度）**：通常 0.7 以上才认为有效，过低会导致误识别。
  - **坐标转换**：OpenCV 识别的是截图内的相对位置，如果不是全屏截图，返回坐标需加上截图起始点的 $X, Y$ 偏移。

### 5.5 解决画面卡顿：零延迟自动适配

在多开或网络延迟（如远程桌面）环境下，代码执行速度远快于画面刷新速度。如果直接循环移动，鼠标会因为“读取到旧位置”而产生来回震荡（转圈圈）。

- **核心逻辑**：引入“临时位置”对比。
- **实现原理**：在执行移动前，记录上一次的 X、Y 坐标。如果当前探测到的坐标与上一次完全相同，说明画面还没刷新，此时程序进入“等待刷新”状态（卡顿计数），而不是盲目再次发送移动指令。
- **补偿机制**：设置一个卡顿阈值（如30次）。如果画面长时间不刷新，强制触发一次微小移动来“激活”状态，防止逻辑死锁。

### 5.6 鼠标偏移的自动学习与修正

由于模拟器、分辨率或不同窗口模式的差异，鼠标的绝对坐标和实际点击点往往存在偏差。

- **解决思路**：**全局变量记录 + 增量学习**。
  - **初始修正**：根据鼠标在屏幕的不同区域（如左上、右下），预设一个初始的偏移值（如 `-70`, `+15` 等）。
  - **动态校准**：程序在“相对移动”校正过程中，如果发现连续多次（如5次）都在向同一个方向微调，说明初始偏移不准。此时自动修改全局偏移变量（加1或减1）。
- **效果**：随着打图次数增多，鼠标会越来越“准”，第一次跳跃的位置会无限接近目标点，减少后续微调次数。

### 5.7 边界与特殊区域处理

当目标点位于窗口边缘时，鼠标容易移出窗口或跳到任务栏。

- **逻辑处理**：在判定坐标小于50或大于屏幕边界时，强制给偏移量加/减一个固定值（如60-70像素），强行把鼠标“拽”回窗口内，再进行细微修正。

### 6.2 打图任务读取

我们采用了**任务追踪栏（快捷栏）读取**的方案，这是目前最快、最稳定的方式。

1. 关键技术：行间距

这是大漠读取多行文本的“深水坑”。

- **操作要领**：
  - 读取前：必须使用 `SetRowGap(0)` 将行间距设为 0，否则多行文本（如任务坐标在第二行）无法连贯读取。
  - 读取后：必须还原为默认值（通常是 1），以免影响其他界面的文字识别。

2. 任务文本解析逻辑

通过 `取出中间文本` 命令，从识别到的字符串中精准提取：**怪物名**、**地图名**、**坐标X**、**坐标Y**。

| **任务关键词类型** | **前缀（Start）**      | **后缀（End）** | **提取目标** |
| ------------------ | ---------------------- | --------------- | ------------ |
| **通缉类**         | `通缉的强盗`           | `目前已逃至`    | 怪物名称     |
| **抢劫类**         | `正在`                 | `手中抢劫`      | 怪物名称     |
| **地图类**         | `逃至`                 | `，据可靠`      | 地图名称     |
| **坐标类**         | （利用数字匹配或分割） | -               | X, Y 坐标    |

一般覆盖以下四种任务语境的解析：

1. **“被官府通缉的强盗 [XXX] 逃至 [地图] [坐标]”**
2. **“商人反应 [XXX] 正在 [地图] [坐标] 拦路抢劫”**
3. **“常匿在 [地图] [坐标] 附近的强盗 [XXX]”**
4. **“正在 [地图] [坐标] 为非作歹的强盗 [XXX]”**



1. **界面判定的封装**：
   - 通过寻找“任务提示”图标来判定任务栏是否打开。
   - 利用相对坐标偏移（基于找到的图标坐标进行计算），解决了任务框被玩家随意拖动或遮挡导致的识别失败问题。
2. **OCR 插件的集成与应用**：
   - 引入了基于飞桨（Paddle）的 OCR 模型，能够识别游戏内非标准系统字体的任务描述。
   - **区域锁定**：通过计算，精确锁定了“任务分类”和“任务详情”两个坐标范围，减少了 OCR 识别干扰（如“追踪本组”等无关文字）。
3. **任务数据的清洗与拆分**：
   - 针对 OCR 识别出的字符串（如“强盗正在化生寺...105, 17”），利用文本处理手段拆分出：**怪物名、地图名、X坐标、Y坐标**。
   - 针对不同任务类型（如“请前往制服”、“被官府通缉的”、“商人反应”）做了多重条件判定，确保数据读取准确。
4. **异常处理流程（闭环逻辑）**：
   - 如果 OCR 读取的结果不符合要求（如坐标为0或名字为空），脚本会自动关闭任务栏并触发下一次尝试，形成了一个自动纠错的逻辑死循环，直到读取成功。

辅助理解：OCR 识别与坐标偏移原理

- **锚点 (Anchor)**：即你代码中的“任务提示”图标。
- **相对偏移 (Relative Offset)**：无论任务框在屏幕哪个位置，详情页相对于锚点的距离是固定的。
- **OCR 区域 (OCR Region)**：锁定这个矩形，只给插件提供这部分图像，识别效率最高。

## 7.跨地图自动寻路与飞行旗联动系统

### 7.1 核心逻辑流程

老师展示的是一套“自适应”的寻路逻辑：

1. **自省（我在哪？）**：通过字库识别左上角地图名和坐标。
2. **判断（要去哪？）**：对比目标地图。如果在当前地图，直接小地图移动；如果不在，寻找最近的“节点”（传送阵）。
3. **执行（怎么去？）**：
   - **跨图**：寻路到传送阵坐标击传送判断地图是否切换成功。
   - **本图**：计算坐标与像素的比例操作小地图点击等待人物位移。



### 7.2 关键技术点分析

**1. 地图坐标与像素的算法转换**![coordinate system transformation，AI 生成](https://encrypted-tbn0.gstatic.com/licensed-image?q=tbn:ANd9GcSuGStQ1Mle3m3O0Bmp3zi50BkeyRA30-4PIqRU5blYfGDSPfpDko_Q1A0OSHayPwz_8lj8dYx-ZdRhneOOcAoeSoz9VJ9VhKGxXCMOk80kmG2iavw)

这是寻路最核心的部分。游戏坐标（如 100, 58）不能直接点击，必须转为小地图上的像素坐标。

- **比例公式**：
  - $X轴比例 = \frac{地图最大X坐标}{小地图图片宽度}$
  - $Y轴比例 = \frac{地图最大Y坐标}{小地图图片高度}$
- **偏移修正**：因为小地图在窗口中的位置不固定，老师通过寻找“寻人”按钮或固定图标，利用偏移量（如 `减3, 减10`）来精确定位小地图的 $0,0$ 坐标原点。

**2. 字库切换的安全性（防止混乱）**

在多线程或频繁调用识字的脚本中，字库序号容易被改乱。老师提供了一个**“临时备份-还原”**的模板：

代码段

```
.子程序 地图_获取当前位置
.局部变量 临时智库序号, 整数型

临时智库序号 ＝ 全局_大漠.取字库序号 ()  ' 1. 备份当前字库
全局_大漠.置字库 (2)                   ' 2. 切换到地图专用字库

' ... 执行识别逻辑 ...

全局_大漠.置字库 (临时智库序号)         ' 3. 退出前还原，防止外部程序识字失败
```

**3. 跨地图的“节点”式寻路**

老师建议初学者不要搞太复杂的自动路径搜索（如 A* 算法），而是采用**“硬拆线路”**的方式。

- **案例：到建邺城**
  - **长安城节点**：寻路到 (538, 10)点击进入江南野外。
  - **江南野外节点**：寻路到 (148, 5)点击进入建邺城。
- **优点**：逻辑简单，调试方便，不容易卡死。



### 7.3 避坑指南（老师的血泪经验）

1. **地图名格式**：尽量使用能稳定识别的字库，避免因为游戏更新导致左上角字符识别偏移。
2. **坐标处理**：使用 `到整数()` 时，注意过滤掉 `[` 或 `]` 等干扰字符。老师演示了即使带括号，`到整数` 也会尝试处理，但最好在逻辑中先进行文本分割。
3. **小地图移动死循环**：点击小地图后，必须加上“寻路等待”逻辑。判断坐标是否在变动，如果坐标不动了且没到目的地，才判定为卡死或需要重试。
4. **鼠标按键习惯**：梦幻西游主交互是**左键**，老师提醒在写脚本时不要和其他游戏（如右键走路的游戏）混淆，导致点击失效。

### 7.5飞行旗

这节课我们深入探讨了**飞行旗（长安、长寿）与飞行符**的自动化逻辑，尤其是针对“小红点缺失”或“不固定坐标”的情况，采用了一套**基于代价（Distance Cost）的冒泡算法**来实现智能就近飞行。

以下是本次课程的核心知识点总结：

**1. 背包界面的动态锁定（防止窗口拖动失效）**

在进行任何飞行操作前，必须确保背包已打开且定位准确。

* **逻辑原理**：不使用死坐标，而是识别背包内的固定标识（如“锦衣”按钮或“道具”文字）。
* **偏移锁定**：根据识别到的标识坐标，反向推算出整个物品栏的矩形范围。这样即使玩家手动拖动了背包窗口，脚本依然能精准找到格子。
* **自动切换**：判断当前是否在“道具”栏，若在“行囊”则自动点击切换。

**2. 飞行旗：智能坐标匹配算法（核心难点）**

针对老区飞行旗点位可能消失（如药店点）或位置偏移的问题，我们放弃了“死点点击”，改用**枚举描点法**。

* **坐标换算**：将游戏内的小地图坐标（如 38, 129）按比例（如 1.5 倍率）换算成相对于屏幕的像素偏移。
* **代价计算（冒泡算法）**：

1. 使用 `FindShape` 或 `FindPicEx` 枚举出当前飞行旗面板上所有的“小红点”。
2. 计算每个红点坐标 $(x1, y1)$ 与目标计算坐标 $(x2, y2)$ 的绝对距离之和：$Cost = |x1 - x2| + |y1 - y2|$。
3. **取最小值**：遍历所有红点，保留 `Cost` 最低的那一个。


* **优点**：即使目标点的小红点消失了，脚本也会自动选择离它最近的一个现有点位飞过去，绝不卡死。

**3. 飞行判定与冗余处理**

* **状态监测**：在点击飞行前，先记录当前的地图名或坐标。点击后，循环检测地图位置是否发生变化。
* **防止连点**：通过 `while` 循环配合 5 秒超时，防止在新区网络波动时，由于面板弹出慢而导致的重复点击或逻辑混乱。

**4. 飞行符：固定逻辑封装**

相比飞行旗，飞行符的逻辑相对简单，主要采用**固定坐标点击**。

* **流程**：右键飞行符 -> 弹出城市列表 -> 点击目标城市（如“长寿村”） -> 监测地图切换。
* **代码规范**：建议使用常量（Const）定义城市名称，防止代码中出现硬编码字符串导致的拼写错误。

**5. 课后练习**

1. **完善旗子分栏**：自行实现“长安旗放第一行、长寿旗放第二行”的分类逻辑，提高识别速度。
2. **扩展地图比例**：尝试计算朱紫国、傲来国的坐标转换比例（寻找 0,0 点与最大点之间的像素差）。
3. **异常捕获**：思考如果飞行旗次数用完了（旗子消失或变色），该如何在代码中添加补给提醒。

## 8.处理动态NPC技巧

处理**动态NPC（车夫、驿站老板）点击**、**对话框自动化**以及**新区遮挡判定**的高级实操技巧。

以下是针对本课内容的详细总结与核心逻辑梳理：

### 8.1 驿站老板（动态NPC）的精确定位与点击

驿站老板（如长安驿站、国境门口）这类NPC是会小范围随机移动的，且容易被玩家、祥瑞或摊位遮挡。

1. **环境清理（屏蔽干扰）**：
   - **屏蔽商铺**：新区摊位满地，首先判定是否存在商铺图标。如果存在，通过 `Alt+H`（或其他快捷键）进行屏蔽。
   - **屏蔽玩家**：同时屏蔽周围玩家和祥瑞，确保NPC的图像特征完整暴露。
2. **特征点抓取**：
   - 采用“红领巾”或其他显著特征点。考虑到NPC有正面和背面，需要准备多张不同角度的素材。
   - **智能判定**：通过 `FindPicEx` 枚举区域内的坐标，若找不到则适当降低匹配度（如从 0.8 降到 0.7），动态寻找点击点。
3. **防超时处理（房卡逻辑）**：
   - 在点击NPC前，先记录当前时间。如果 5 秒内没有弹出对话框，说明点击失败（可能点到了祥瑞或路人），需重新寻找点击。

### 8.2 对话框逻辑与字库管理

一旦点击成功，如何稳定地选择“我要去大唐国境”这类选项是自动化的关键。

1. **对话框判定（封装）**：
   - 封装 `界面判定_是否存在对话框` 子程序。
   - **关键点**：在进入识字逻辑前，必须备份当前字库索引，识别完后立刻还原（`置字库`）。防止因为子程序内部修改了字库，导致外部主逻辑（如寻找强盗）失效。
2. **选项识别与点击**：
   - 识别“我要去”等关键字。
   - **点击偏移**：识字返回的是文字左上角坐标，点击时需向下、向右进行适当偏移（如 `+6, +8`），确保点在文字中心区域。

### 8.3  跨地图流程控制

老师展示了如何通过**执行流程（状态机）**来管理长距离位移：

- **第一阶段**：寻路至驿站老板坐标点（如 280, 40）。
- **第二阶段**：检测到抵达后，开启屏蔽逻辑并寻找NPC。
- **第三阶段**：对话选择目的地，并监测地图名变化。
- **异常处理**：若点击后地图名长时间未改变，说明传送失败，逻辑自动跳回第一步重新开始。

### 8.4 应对强盗遮挡与随机刷新（预告及核心思路）

虽然这部分在演示中受限于服务器环境（老区没怪），但老师给出了核心算法逻辑：

- **计算强盗位置**：强盗刷新坐标虽随机，但有迹可循。通过获取任务列表中的坐标，结合地图比例计算屏幕像素点。
- **遮挡判定**：
  - 当点击强盗后，如果没进入战斗（通过战斗界面标志判断），说明被乌铁盒、奖励箱或NPC遮挡。
  - **处理方案**：计算强盗坐标周围的小范围偏移点（如上下左右偏 5 像素），循环尝试点击。

课后避坑要点：

1. **智库还原**：一定要在子程序出口处还原字库，这是最容易导致脚本莫名其妙不识字的“暗坑”。
2. **VNC延迟**：针对远程云手机或虚拟机，画面刷新慢。识字判定建议加入小循环（如 5 次判定），给画面一点缓存时间。
3. **对话框防卡**：如果用户手动点开了对话框，脚本要有“房卡”逻辑去主动接管，而不是强行去寻找关闭按钮。

## 9.地图坐标换算

这节课我们深入讲解了如何通过**地图坐标换算**，在不依赖图像识别的情况下，精准计算并点击游戏画面中的 NPC 或怪物（如打图任务中的强盗）。

这种方法对于处理**怪物被遮挡**、**模型重叠**或**随机位置刷新**的情况具有极高的稳定性和效率，是回合制游戏自动化的核心技术。



### 9.1 核心原理：屏幕与地图的比例换算

所有的 2D 或 2.5D 回合制游戏（梦幻、大话等）都存在一个**屏幕坐标**与**地图坐标**的比例因子。

- **比例因子**：根据老师的实测，在 800×600 分辨率下，比例系数约为 **20.5**。即地图坐标每变动 1 个单位，屏幕像素大约变动 20.5 个像素。
- **基准点**：
  - **非边界状态**：人物始终位于屏幕正中心（如 400, 300）。
  - **边界状态**：当人物靠近地图边缘（X < 19 或 Y < 15 左右），人物不再固定在中心，屏幕开始停止滚动。



### 9.2 三种核心算法场景

计算公式根据人物是否在屏幕中心分为三种逻辑：

A. 人物在屏幕中心（通用场景）

当人物坐标远离边界时，NPC 的屏幕坐标取决于它与人物的坐标差。

- **X 坐标**：$400 + (NPC_X - 人物_X) \times 20.5$
- **Y 坐标**：$300 + (人物_Y - NPC_Y) \times 20.5$（注意：游戏 Y 轴通常与屏幕 Y 轴方向相反）

B. 人物处于左/上边界（坐标接近 0）

当人物坐标极小时，屏幕锁死在地图边缘，此时 NPC 坐标直接由其绝对位置决定。

- **算法**：$NPC_{屏幕} = NPC_{地图} \times 20.5$
- **应用**：直接通过 NPC 的地图坐标乘以比例，从屏幕左上角（或偏移量）开始计算。

C. 人物处于右/下边界（坐标接近最大值）

这是最容易出错的地方，因为需要从屏幕的最大宽度/高度反向推算。

- **X 坐标**：$800 - (地图最大_X - NPC_X) \times 20.5$
- **Y 坐标**：通过地图最大 Y 轴减去 NPC 的 Y 轴距离来计算。



### 9.3 实战代码逻辑梳理

JavaScript

```
// 伪代码参考
function 点击NPC(目标坐标) {
    let 当前位置 = 获取自身地图位置();
    let 比例 = 20.5;
    let 点击X, 点击Y;

    if (当前位置.X > 19 && 当前位置.X < (地图最大X - 19)) {
        // 1. 人物在中心
        点击X = 400 + (目标坐标.X - 当前位置.X) * 比例;
    } else if (当前位置.X <= 19) {
        // 2. 左边界
        点击X = 目标坐标.X * 比例;
    } else {
        // 3. 右边界
        点击X = 800 - (地图最大X - 目标坐标.X) * 比例;
    }
    
    // Y轴同理处理...
    
    MouseClick(点击X, 点击Y);
}
```

------

### 9.4  细节优化与避坑指南

1. **绝对值与偏移**：在计算边界距离时，建议使用 `abs()` 取绝对值，防止出现负数点击导致鼠标飞出游戏窗口。
2. **微调边框**：由于游戏窗口存在标题栏和边框（如底部的任务栏），屏幕高度不一定是纯粹的 600。老师建议使用 **595** 或 **580** 作为基准进行微调，防止点到怪物的“脚尖”以下。
3. **对话框判定**：点击完成后，必须紧跟一个对话框判定（如识别“我要去...”或“进入战斗”），如果没有触发，则进行小范围偏移随机点击，解决怪物被箱子或 NPC 遮挡的问题。
4. **比例动态化**：不同地图或不同分辨率下，20.5 这个数值可能有微小差异。建议针对每个大地图（如大唐国境、长安城）做一套配置数据。

总结

通过这套算法，你的脚本将拥有一种“上帝视角”。即使屏幕上叠了 100 个玩家或装饰物，只要你知道强盗的坐标，就能一击必中。

## 10.**强盗被遮挡时的自动偏移探测点击**

这节课我们重点讲解了**强盗（NPC）被遮挡或重叠时的进阶处理逻辑**。在实际挂机打图过程中，强盗经常会被乌铁盒、奖励箱或者人物自身模型挡住，导致普通点击失效。

以下是本次课程的核心技术总结：

------

### 10.1点击流程与“寻路静止”判定

在点击强盗之前，必须确保人物处于静止状态。

- **逻辑陷阱**：如果人物正在向强盗移动，脚本就开始疯狂点击坐标，会导致人物在目标点附近来回“抽搐”，甚至点偏。
- **解决方案**：在点击函数前加入一个 `地图寻路判定`。只有当寻路结束、人物坐标不再变动时，才执行点击动作。这虽然多花了几百毫秒，但极大地提高了点击的**准确率**和**稳定性**。

### 10.2冗余点击与 CTRL 穿透机制（核心算法）

针对被遮挡的情况，我们设计了一套“利大于弊”的选择性触发逻辑：

- **第一阶段：常规点击**（快速、自然）：

  直接根据坐标点击强盗。在 90% 的情况下，这是最快的触发方式，不会有额外的按键动作。

- **第二阶段：计数判定**：

  如果连续点击 3 次（或自定义次数）后，依然没有检测到对话框弹出，说明大概率被遮挡了。

- **第三阶段：触发穿透按键（CTRL/ALT）**：

  当 `点击次数 >= 3` 时，脚本自动按住 `CTRL` 键（或模拟向上翻页）。此时，游戏会屏蔽掉遮挡物，直接暴露出强盗的点击层。

### 10.3 CTRL 模式下的特征识别

一旦按住 `CTRL` 触发了穿透模式，强盗会以特定的列表或气泡形式出现。

- **动态找字**：我们从任务详情中读取“怪物名字”（例如：强盗、无名鬼），在弹出的穿透列表区域内进行 `FindStr`（找字）。
- **自动修正**：找到怪物名字后，计算偏移坐标并实施点击。由于这是在穿透模式下操作，点击的必定是怪物本身，彻底解决了“点错”问题。

## 11.贼王处理机制

这节课我们进入了整个打图流程中最核心、也是技术细节最复杂的难点——**贼王处理机制**。贼王不同于普通强盗，其随机性强、地图大、且极易发生模型叠压。

以下是针对贼王处理的**三大核心步骤**及技术实现总结：

### 1. 傻瓜式巡路寻敌（解决大地图盲区）

由于贼王可能出现在房间的任何角落（如酒店二楼、国子监等大地图），传统的死坐标点击极易失败。

- **策略**：采用“密集图片枚举寻路”。
- **实现**：预先在房间内截取 8-10 张标志性建筑的特征图（如楼梯口、灯笼处、柜台旁）。脚本进入房间后，按照图片序列依次查找并移动，形成一个覆盖全图的“巡回路线”。
- **优势**：简单粗暴，无需计算复杂的动态坐标，只要人物在动，就能扫描到视野内的所有 NPC。

### 2. 多重叠压下的“贼王遍历”算法（核心难点）

这是解决“五六个 NPC 叠在一起”问题的关键。

- **穿透识别**：利用 `OpenCV` 找图获取当前视野内所有疑似贼王的坐标点，并将其存入数组。
- **循环检测机制**：
  1. **按住 CTRL 键**：强制穿透人物/物品遮挡，弹出目标列表。
  2. **点击识别**：依次点击数组中的每一个贼王坐标。
  3. **对话框判定**：每点开一个列表，立即切换到专属的“对话框字库”，识别其中的“怪物名称”是否与当前任务目标匹配。
  4. **排除干扰**：如果识别出的名字不是目标贼王，立即点击右键取消对话框（或点击向上翻页关闭列表），继续点击下一个坐标点。

### 3. 字库切换与坐标运算

- **字库动态切换**：为了提高识别效率，进入对话框识别流程时，临时将系统字库切换为“对话框专用字库”，识别结束后再掷回。这样可以极大减少字符干扰。
- **坐标偏移修正**：贼王的名字高度通常是固定的。在 `OpenCV` 捕捉到身体中心点后，向上加减固定偏移量（如 `Y - 26`）来定位名字区域，确保护标点击在有效区域。

------

### 技术要点提示（避坑指南）：

1. **数组越界保护**：在分割坐标字符串时，**必须**先判断 `取数组成员数() > 1`。否则一旦没找到贼王，脚本会因为访问空数组直接闪退。
2. **鼠标复原**：每次点击完列表或非目标 NPC 后，一定要把鼠标移动到视野边缘（复原），防止遮挡下一步的找图。
3. **OpenCV 精度**：贼王识别的精度建议设定在 `0.7 - 0.75` 之间。太高会因为人物动作变化漏掉，太低会把普通玩家当成贼王。



## 12.贼王进阶处理

### 一、 知识点概述

本章重点讲解了贼王（野外与房间内）的定位识别、拉近算法、以及集成化 OCR 验证逻辑（包括成语、四小人、漂浮字等）。通过“巡逻-识别-拉近-触发”的标准化闭环，解决 NPC 重叠和位置偏移问题。

------

### 二、 贼王任务处理逻辑 (ZeiWang Logic)

1. 定位分析：野外 vs 房间内

- **野外贼王**：游戏给出的坐标仅为**模糊周边坐标**。抵达后需执行局部扫描识别。
- **房间内贼王**：对于大场景（如奥莱客栈 2 楼），需设计固定巡逻线路（Checkpoint 模式）以覆盖视觉盲区。对于小场景（如杂货店、当铺），可直接按野外识别逻辑处理。

2. 核心算法：拉近与遍历 (Distance & Traversal)

由于贼王位置具有不确定性且容易被玩家模型遮挡，采用以下双层逻辑：

- **穿透枚举**：利用 `OpenCV` 获取视野内所有疑似 NPC 的坐标，按住 `CTRL` 键进行穿透点击，避免遮挡。
- **拉近逻辑 (Proximity Implementation)**：
  - **首次识别**：确定目标为当前任务贼王。
  - **位置修正**：若距离过远无法触发对话，脚本点击 NPC 附近的偏移坐标（如 `X-100, Y+65`）执行移动命令。
  - **二次触发**：移动到位后，将 `是否拉近过` 标记为 `True`，重新执行点击触发对话。

### 三、 高性能 OCR 集成验证 (OCR Validation)

1. 架构设计

采用本地化 DLL 方式，集成轻量化深度学习模型，支持秒级识别，避免全屏识别造成的资源浪费。

2. 初始化与调用

- **初始化**：全局仅需初始化一次，载入模型路径。
- **基于图片识别**：所有坐标均基于截图的 `(0,0)` 原点，而非屏幕坐标，极大提升了多开兼容性。

3. 验证类型详解

| **验证类型** | **标识码** | **逻辑要点**                                            |
| ------------ | ---------- | ------------------------------------------------------- |
| **四小人**   | 2          | 返回矩形中心点坐标，直接模拟点击即可。                  |
| **成语验证** | 1          | 采用 `题目:答案` 分割模式。识别完整成语及单字精准坐标。 |
| **漂浮字**   | 3          | 动态坐标，需加入循环实时识别，直到图层消失。            |

4. OCR 返回数据解析 (以成语为例)

- **格式**：`题目:挥金如土 | 挥,167,56,0.9,1 | 金, ...`
- **尾缀含义**：末尾的 `1` 代表上排字符，`2` 代表下排字符。
- **坑点处理**：成语验证每点对一个字，图层内容会刷新，脚本需执行“点击-重识”的循环，直至验证通过。

------

### 四、 核心技术细节与“避坑”指南

1. **坐标偏移 (Offset)**：贼王点击坐标需进行视觉补偿（如 `Y轴 - 50`），确保点击在名字或头顶上方，避免点到地面。
2. **视觉防呆 (Fail-safe)**：
   - 在识别到 NPC 后，先移动鼠标至目标位置，判断是否弹出对话框。
   - 若对话框未弹出，通过 `CTRL` 穿透重试，防止因模型重叠导致的误点。
3. **OCR 性能优化**：
   - **严禁全屏识别**：必须针对验证码区域进行局部截图，将耗时压缩至 40-100ms。

## 14.自动登录



### 一、 流程设计概览 (Workflow Overview)

自动登录流程是一个多线程、多窗口协作的过程，主要包含：**INI配置读取**、**模拟器将军令获取**、**游戏窗口绑定**三个核心阶段。

### 1. 自动化逻辑路径

1. **数据初始化**：从 INI 文件读取账号、密码、大区、角色索引及关联的将军令 ID。
2. **前置验证获取**：判断是否需要将军令。若需要，优先启动模拟器获取动态验证码。
3. **时效性策略**：将军令具备约 5 分钟的有效时长，建议“先获取，后上号”，以规避登录过程中的操作超时。
4. **环境切换**：获取验证码后，需从模拟器句柄切换回游戏窗口句柄。

------

### 二、 核心模块实现 (Core Modules)

### 1. 账号信息读取与封装

利用 INI 结构化存储账号数据，通过字符串分割技术（Split）实现数据归类。

Ini, TOML

```

1 = 账号----密码----角色索引----大区----将军令索引
```

**技术要点：**

- **角色索引**：用于处理单账号多角色选择。
- **将军令索引**：对应多开模拟器中的特定实例 ID，确保账号与验证器一对一绑定。

### 2. 模拟器控制与将军令识别

通过雷电模拟器指令集（LDConsole）或相关模块，实现对特定模拟器窗口的控制。

- **窗口绑定**：获取模拟器句柄（HWND）并进行后台绑定。
- **图像识别 (OCR)**：
  - **方案 A (字库)**：通过大漠制作数字字库（0-9），需注意背景偏色导致的识别误差。
  - **方案 B (本地模型)**：利用轻量化 OCR 插件直接识别 6 位数字，稳定性更高。
- **容错处理**：若识别结果长度不等于 6 位，则模拟点击刷新按钮，并延迟 5 秒后重试。

### 3. 句柄环境切换 (Context Switching)

这是登录流程中最关键的步骤。脚本需在“模拟器窗口”与“游戏 VNC 窗口”之间执行绑定切换。

### 三、 代码逻辑伪代码 (Technical Pseudo-code)

Python

```python
# 自动登录核心逻辑伪代码
def auto_login_process(index):
    # 1. 读取INI配置
    account_info = read_ini_config(f"窗口{index}")
    
    # 2. 判断将军令需求
    if account_info.need_token:
        # 切换句柄：绑定到模拟器
        bind_emulator(account_info.token_id)

            token = ocr_get_token() # 识别6位数字
            if len(token) == 6:
                save_to_global(token) # 存入全局变量供登录逻辑调用
                break
            click_refresh_button() # 识别失败则刷新

        unbind_emulator()
        bind_game_window(index)
    
    # 4. 执行后续登录界面判定
    enter_login_stage()
```

### 四、 核心技术细节与“避坑”指南

1. **将军令时效利用**：将军令刷新后会有一定的“宽容期”，即使模拟器显示的倒计时结束，旧代码在短时间内依然有效。利用这 5 分钟的冗余，可以极大地提高脚本上号的成功率。
2. **模拟器绑定模式**：模拟器背景绑定建议使用 `gdi` 或 `dx.graphic.2d` 模式，确保在窗口被遮挡时依然能稳定获取图像。
3. **字库精细度**：如果采用字库识别，必须针对模拟器界面的每个数字进行多点颜色抓取。由于将军令界面可能存在抗锯齿边缘，建议设置色差范围在 `10%-20%` 之间。
4. **CMD 控制指令**：推荐使用雷电控制台指令（ldconsole.exe）直接通过索引操作模拟器（如：`launch` 启动、`quit` 关闭），比纯图像寻找图标更可靠。

## 15.任务超时监控与游戏掉线自动重连

自动化脚本运行过程中，网络波动、游戏卡死或意外弹窗会导致脚本陷入“死循环”或“原地踏步”。通过建立一套独立的**线程监控体系**，可以像“监工”一样，在主程序发生意外时强制干预，重启流程。

------

### 一、 超时监控系统架构 (Timeout Monitoring Architecture)

我们采用“**双线程协作**”模式：

1. **脚本主线程 (Master Thread)**：负责执行登录、打图、任务分配等逻辑。
2. **监控辅助线程 (Monitor Thread)**：独立运行，负责倒计时。当倒计时归零时，判定为异常，执行强制重启。

------

### 二、 核心实现逻辑

1. 动态倒计时机制 (Dynamic Countdown)

监控线程并不是死板的固定时间，而是通过**全局变量**进行动态交互：

- **重置 (Reset)**：主线程每完成一个关键动作（如：成功进入战斗、切换地图、领到任务），就将全局变量 `超时时间` 重置为一个较大的数值（如 1000 秒）。
- **递减 (Decrement)**：监控线程每秒对 `超时时间` 执行 `-1` 操作。
- **触发 (Trigger)**：一旦变量值 `< 0`，监控线程立即接管控制权，判定主线程已卡死。

2. 强制关闭游戏进程

当确认超时后，传统的点击“退出按钮”可能失效（界面卡死）。我们采用 **Windows 命令级强制终止**：

- **技术手段**：通过 VBS 或 `taskkill` 命令强制结束 `my.exe`（梦幻西游进程）。

- **实现代码示例**：

  `taskkill /F /IM 梦幻.exe` （其中 `/F` 表示强制，`/IM` 表示映像名称）。

3. 全流程自动重启 (Full Auto-Restart)

关闭卡死的进程后，监控线程会执行以下自动化复原步骤：

1. **销毁旧线程**：清理掉当前卡死的主线程句柄。
2. **重置状态**：将脚本内部状态标识符设为“已停止”。
3. **重新启动**：自动调用 `主引擎入口`，重新执行：**启动模拟器 -> 获取将军令 -> 输入帐号密码 -> 登录 -> 继续任务**。

------

### 三、 技术细节与避坑指南

1. 线程挂起与清理 (Thread Suspension)

在执行重启逻辑前，务必先**挂起 (Suspend)** 原主线程，并执行 `鼠标左键弹起` 等初始化操作。防止重启过程中，旧线程残留的“按键按下”状态干扰新进程。

2. 超时阈值的设定建议

- **登录阶段**：建议设为 300-500 秒（考虑到将军令获取和排队）。
- **常规打图**：建议设为 120-180 秒（正常跑路+打怪的时间）。
- **验证码阶段**：如果检测到弹出了验证码，应通过代码动态延长超时时间，给人工或 OCR 留出处理余地。

3. 线程安全 (Thread Safety)

由于两个线程同时读写同一个变量（`超时时间`），在易语言或 Python 中，建议对该变量的操作加装 **许可证 (Critical Section)**，防止因读写冲突导致的脚本崩溃。

------

### 四、 总结与后续

本节课实现的“超时重启”逻辑，是让脚本具备“**无人值守**”能力的最后一块拼图。

### 四、 跨账号“自动复原”逻辑 (Cross-account Recovery)

1. 缓存与镜像分离

- **`Task_Flow` (实时流)**：随着任务完成而不断缩减，用于断点恢复。
- **`Task_Mirror` (原始镜像)**：存储用户最初设定的完整任务链。

2. 换号重置触发

1. **检测空链**：识别到当前账号任务已全部结束。
2. **镜像复位**：将 `Task_Mirror` 的内容重新拷贝给 `Task_Flow`。
3. **新号启动**：下一个账号登录后，读取的是被重置后的完整任务链。

### 五、 关键避坑指南 (Troubleshooting)

1. **索引偏移问题**：易语言计次循环索引从 `1` 开始，而超级列表框索引从 `0` 开始。在遍历删除时务必注意 `Index - 1` 的对应关系。
2. **UI 线程安全**：在多线程脚本中，直接操作 UI 控件极易崩溃。**强烈建议使用标签反馈或消息通知机制**进行 UI 更新。
3. **配置写入时机**：不要仅在任务全部结束时写配置。应在“每一个子任务”结束时立即同步本地文件，以应对随时可能发生的掉线或崩溃。
4. **路径纠错**：使用 `取运行目录()` 获取相对路径。避免在系统重装或目录移动后，因路径失效导致配置无法读取。

## **18.任务实战（押镖任务）**

### 一、 核心逻辑：节点算法（无坐标寻路）

在押镖过程中，系统不显示当前坐标，且小地图的小红点会随移动产生巨大漂移。你采取的**地标节点法**是目前最稳健的方案：

- **逻辑原理**：将漫长的路径拆解为多个“节点”（例如：长安 -> 国境入口 -> 境外入口）。
- **识别机制**：
  - **地标图片识别**：通过 `open CV` 或 `大漠找图` 识别地图边缘的标志性建筑或地名（如大唐国境的“唐”字）。
  - **执行循环**：脚本在移动过程中不断检测“下一站”的特征图。只要没看到特征图，就判定为“未抵达”，继续点击寻路坐标；一旦识别到特征图，立即跳转到下一个节点。
- **卡屏检测**：利用大漠的卡屏函数，判断屏幕是否长时间停止不动。如果地图名变了且屏幕不动，则判定为由于传送导致的寻路结束。

------

### 二、 战斗逻辑的“秒返回”与衔接

押镖中的随机战斗是最大的干扰项。你展示的**外层拦截法**非常精妙：

- **战斗处理**：将战斗逻辑（如点选“四小龙”、点击“自动”）封装在最外层的主循环中。
- **递归返回**：在所有移动、等待的底层函数中，实时监控 `是否在战斗`。一旦进入战斗，底层函数立即返回 `False` 或特定状态码，逐层跳出到主循环。
- **状态重置**：战斗结束后，脚本并不尝试从中断点继续，而是**重新读取任务状态**。通过动态获取当前地图位置，重新触发寻路。这种“重置式衔接”极大地降低了逻辑死锁的概率。

------

### 三、 死亡异常与取消任务处理

针对押镖失败（死亡）后的逻辑闭环：

- **轮回司判定**：在主循环中加入“白无常”或“轮回司”场景判定。一旦检测到死亡，自动寻路回长安。
- **强行取消**：通过脚本走到镖局门口，对话NPC取消当前已失败的任务。
- **状态回归**：任务取消后，任务列表变空。脚本会自动进入“读取任务 -> 没任务 -> 接任务”的初始循环，实现无人值守。

------

### 四、 技术细节总结表

| **难点**       | **解决方案**              | **技术关键词**             |
| -------------- | ------------------------- | -------------------------- |
| **无坐标**     | 特征地标+节点算法         | `找图ES`, `节点配置`       |
| **随机战斗**   | 全域监控+底层强制返回     | `是否在战斗`, `主循环拦截` |
| **寻路超时**   | 卡屏检测+实时地图名抓取   | `大漠卡屏检测`, `文字识别` |
| **跨地图移动** | 边界坐标点击+场景切换判定 | `坐标偏移`, `场景ID检测`   |

## 19.任务实战

### 1. 自动转图流程拆解

整个功能被划分为几个核心阶段，每个阶段都有明确的判定条件：

- **场景判定与移动**：首先获取当前地图名（如“长安城”），判断是否在仓库管理员（347, 243）或宝石商人（496, 144）坐标附近。若不在，使用飞行服飞往目标点。
- **交互与界面打开**：通过坐标偏移点击NPC，并实时监测是否弹出对话框。利用字库识别“仓库”等关键词，精确点击并进入仓库界面。
- **背包状态监测**：通过识别背包中的“空格子”数量来判断是否已满。你这里分享了一个很有趣的技巧：**截取带边框的格子图**，利用四角颜色不一防止被弹幕插件当成透明色处理，从而准确统计空格。
- **批量操作逻辑**：
  - **取图**：在仓库界面利用`找图`功能枚举所有宝图坐标，通过右键循环点击将宝图存入背包，直至背包满或仓库无图。
  - **给予**：自动打开好友列表，通过查找“小房子”图标定位仓库号分组，右键点击“给予”，并控制每次点击3张图后点击确定，提高效率。

### 2. 关键代码逻辑与技巧

- **动态等待（非死延时）**：在点击操作后，使用循环判断界面是否出现，并在5秒内成功则立即跳出。这比固定延时更科学，既能应对服务器卡顿，又能在顺畅时极大提高脚本速度。
- **优先级判定**：在主循环中先判定“给予面板”是否打开。如果面板已打开，直接进入给予逻辑；如果没打开，再判定仓库界面。这种“从内向外”的判定逻辑增强了脚本的容错性，防止用户手动操作干扰。
- **坐标偏移与锁死**：通过识别到一个固定图标（如好友面板标题），计算出相对偏移坐标（X+83, Y+50），从而实现“窗口拖到哪都能点准”的动态定位。

### 3. 常见问题处理

- **杂音与环境干扰**：通过禁麦、调整风扇等方式保证教学质量。
- **位置微调**：讲解了坐标像素的换算关系（小地图1坐标 ≈ 屏幕20像素），方便新手在遇到点击偏离时进行修正。
- **状态闭环**：任务完成后，脚本会自动关闭所有面板（给予框、属性框、列表框），然后重新寻找目标NPC，形成了完美的逻辑闭环。

------

为了让大家更清晰地理解这一整套逻辑，我们可以看这张流程图：

1. **开始** -> **检测位置**（不在长安则飞行）。
2. **移动至NPC** -> **点击NPC触发对话**。
3. **判断背包**（若满则去转图，若空则去取图）。
4. **取图/转图循环**。
5. **完成单次任务** -> **清理界面** -> **回到步骤1**。

## 20.卡密系统搭建

### 一、 BSPAP 网络验证系统架构

该系统是一个典型的 **B/S 架构**（浏览器/服务器），它将脚本的控制权由“本地”转移到了“云端”。

### 1. 核心架构组成

- **管理后台**：由作者使用，用于管理软件版本、生成卡密、查看用户数据、设置公告。
- **代理商后台**：分发给分销商，用于提卡和管理其名下的客户。
- **用户客户端（脚本）**：集成官方 SDK，通过加密 API 与服务器通信。
- **数据库 (MySQL)**：存储所有的用户信息、卡密状态和机器码绑定记录。

------

### 二、 服务器环境搭建 (Environment Setup)

你演示了在 Windows 虚拟机上使用**宝塔面板 (BT Panel)** 快速部署环境，这是目前最高效的方式。

1. 环境套件要求 (LAMP/LNMP)

- **Web服务器**：Apache (或 Nginx)
- **数据库**：MySQL 5.7 (存储效率高且稳定)
- **脚本语言**：PHP 7.0 (BSPAP 运行的基础环境)
- **管理工具**：phpMyAdmin (用于手动维护数据库)

2. 安装逻辑流程

1. **域名/IP 授权**：在 BSPAP 官网将你的服务器 IP 进行授权。
2. **创建网站与数据库**：在宝塔中新建站点，记住生成的数据库名、用户名及密码。
3. **上传源码**：将验证系统的 PHP 源码上传至网站根目录。
4. **初始化安装**：访问 `http://服务器IP/install`，填写数据库信息完成配置。

------

### 三、 脚本集成与联调 (Integration)

在易语言端，通过调用官方提供的模块（.ec）或源码，实现与云端数据的交互。

1. 初始化关键参数

要确保脚本能连上服务器，必须保证以下参数在后台与代码中完全一致：

- **服务器地址**：你的网站 URL（如 `http://1.2.3.4/api.php`）。
- **通信密码**：用于 API 请求的基础加密。
- **通信认证 Key (DES/AES)**：确保发包数据不被轻易拦截抓取。

2. 核心功能实现逻辑

- **公告获取**：每次启动脚本时自动拉取云端公告。

- **卡密登录**：用户输入卡密 -> 脚本发送机器码+卡密 -> 服务器判定余额、绑定状态 -> 返回登录成功 Token。

- **云端变量（重要防破解）**：

  > **核心提示**：不要在本地脚本里写死逻辑。将关键的坐标数据、配置参数放在“软件自定义常量”中。只有登录成功后，脚本才从云端下载这些数据。如果用户暴力破解掉登录框，由于拿不到云端参数，脚本依然无法运行。

 **VMP (VMProtect)** 加壳以及 **滴水论坛** 的防护方案。

```
打图任务执行流程示例
展示状态机在一个while循环中的工作方式
"""

class DaotuTask:
    """打图任务类"""
    
    def __init__(self, window_index):
        self.window_index = window_index  # 窗口索引
        self.current_state = "读取当前任务"  # 当前状态
        self.task_info = None  # 任务信息(字典)
        
        # 状态列表(定义所有可能的状态)
        self.states = [
            "读取当前任务",      # 状态1
            "接取任务",          # 状态2
            "选择传送方式",      # 状态3
            "跨地图导航",        # 状态4
            "同地图寻路",        # 状态5
            "点击强盗NPC",       # 状态6
            "战斗循环",          # 状态7
            "任务完成",          # 状态8
        ]
    
    def run(self):
        """任务主循环 - 只有一个while!"""
        print(f"🚀 窗口{self.window_index} 打图任务开始...")
        
        # ============================================================
        # 核心: 只有一个while循环
        # ============================================================
        while True:  # 👈 只有一个while!
            
            try:
                # ====================================================
                # 状态机: 根据current_state执行对应逻辑
                # ====================================================
                
                # ----------------------------------------
                # 状态1: 读取当前任务
                # ----------------------------------------
                if self.current_state == "读取当前任务":
                    print(f"📍 当前状态: {self.current_state}")
                    
                    # 尝试读取任务(从任务追踪栏或任务面板)
                    task = self._read_task_from_ui()
                    
                    if task is not None:
                        # 👉 任务读取成功,保存任务信息

                        print(f"   怪物名: {task['monster_name']}")
                        print(f"   地图名: {task['map_name']}")
                        print(f"   坐标: ({task['x']}, {task['y']})")
                        
                        # 👉 切换到下一个状态: 选择传送方式
                        self.current_state = "选择传送方式"
                    else:
                        # 👉 没有任务,需要接取任务
                        print("❌ 当前没有任务,需要接取任务")
                        self.current_state = "接取任务"
                
                # ----------------------------------------
                # 状态2: 接取任务
                # ----------------------------------------
                elif self.current_state == "接取任务":
                    print(f"📍 当前状态: {self.current_state}")
                    
                    # 1. 屏蔽其他玩家
                    self._shield_other_players()
                    
                    # 2. 寻路到店小二
                    self._move_to_npc("店小二")

                    if self._click_npc_and_talk("店小二"):
                        # 4. 接取任务
                        print("📋 正在接取打图任务...")
                        time.sleep(2)
                        
                        # 5. 返回状态1重新读取任务
                        self.current_state = "读取当前任务"
                    else:
                        # 点击失败,继续尝试
                        print("⚠️  点击NPC失败,重试...")
                
                # ----------------------------------------
                # 状态3: 选择传送方式
                # ----------------------------------------
                elif self.current_state == "选择传送方式":
```
